# กระบวนการวางโครงเรื่อง (Story Planning Workflow)

## หลักการชี้นำ (Guiding Principles)

1. **TODO.md คือแหล่งอ้างอิงความจริงสูงสุด:** งานทั้งหมดต้องถูกบันทึกและติดตามใน `TODO.md`
2. **โครงสร้างเรื่องต้องชัดเจน:** การเปลี่ยนแปลงโครงสร้างพล็อตหรือตัวละครต้องระบุใน `narrative-framework.md` ก่อนเริ่มปรับเนื้อหา
3. **ตรวจสอบโครงสร้างก่อนขยายความ (Structure-First):** รัน Tool `validate_story_structure` เพื่อเช็คตรรกะของเรื่องก่อนเสมอ
4. **ความสอดคล้องของตัวละครและปม:** ต้องยืนยันความเชื่อมโยงผ่าน Tool `extract_characters` และ `extract_conflicts` ในทุกเฟส
5. **ประสบการณ์ผู้อ่านต้องมาก่อน:** ทุกการตัดสินใจต้องให้ความสำคัญกับความสมเหตุสมผลและอารมณ์ร่วมของผู้อ่าน
6. **เน้นคำสั่งแบบ Non-Interactive:** เลือกใช้คำสั่งที่แสดงผลลัพธ์จบในตัวเพื่อความรวดเร็วในการตรวจสอบพล็อต

---

## เวิร์กโฟลว์ของงาน (Task Workflow)

งานทั้งหมดต้องปฏิบัติตามวงจรชีวิตที่เคร่งครัดดังนี้:

### ขั้นตอนการทำงานมาตรฐาน (Standard Task Workflow)

1. **เลือกงาน (Select Task):** เลือกงานถัดไปจาก `TODO.md` ตามลำดับความสำคัญ โดยดูจาก **Type** และ **Label**
2. **ทำเครื่องหมายว่ากำลังดำเนินการ:** แก้ไขไฟล์ `TODO.md` และเปลี่ยนสถานะงานจาก `[ ]` เป็น `[~]`
3. **วิเคราะห์โครงสร้าง (Red Phase):**
   - เขียนร่างเนื้อหาดิบหรือโครงเหตุการณ์
   - รัน Tool `analyze_story` และ `validate_story_structure` เพื่อหาจุดบกพร่อง (Plot Holes)
   - ยืนยันว่าโครงสร้างยัง "พัง" หรือมีจุดขัดแย้งก่อนเริ่มแก้ไข
4. **แก้ไขเนื้อหาให้ผ่านเกณฑ์ (Green Phase):**
   - ปรับปรุงเนื้อหาจนกว่าโครงสร้างจะถูกต้องและผ่านการ Validate (`success: true`)
   - รัน Tool ซ้ำเพื่อยืนยันผล
5. **ขยายความและเชื่อมโยง (Refactor):**
   - ใช้ Tool `build_relationship_graph` เพื่อเพิ่มความลึกของความสัมพันธ์
   - ปรับปรุงบทบรรยายโดยไม่ให้เสียโครงสร้างหลักที่ผ่านการตรวจสอบแล้ว
6. **ตรวจสอบความครอบคลุม (Verify Coverage):** ยืนยันว่าองค์ประกอบหลัก (ตัวละคร, ปม, จุดคลี่คลาย) ถูกบันทึกลงใน Graph ครบถ้วน
7. **Commit การเปลี่ยนแปลง:**
   - Stage ไฟล์ที่เกี่ยวข้อง
   - ระบุ Commit Message ตามรูปแบบที่กำหนด เช่น `feat(plot): Add inciting incident for character A`
8. **แนบสรุปงานด้วย Git Notes:**
   - **ขั้นตอน 8.1:** ดึง Commit Hash ล่าสุด
   - **ขั้นตอน 8.2:** สร้างสรุปงาน (ชื่อเรื่อง, จุดที่ปรับเปลี่ยน, ผลกระทบต่อตัวละคร, และเหตุผลที่ต้องปรับปมนี้)
   - **ขั้นตอน 8.3:** แนบ Note ด้วยคำสั่ง `git notes add -m "<note content>" <commit_hash>`
9. **บันทึก Commit SHA ของงาน:**
   - อัปเดตสถานะใน `TODO.md` จาก `[~]` เป็น `[x]` และแนบ SHA 7 ตัวแรกต่อท้ายบรรทัด

### รูปแบบการบันทึกใน TODO.md

รายการงานใน `TODO.md` ต้องระบุ **Type** และ **Label** ตามรูปแบบนี้:

```markdown
### Phase 1: Setup & World Building
- [x] `type:feat` `label:core` Define world laws and magic system (sha: a1b2c3d)
- [~] `type:plot` `label:high-priority` Create inciting incident for Protagonist A
- [ ] `type:arc` `label:character` Establish backstories for the main trio
```

### โปรโตคอลการตรวจสอบเมื่อจบเฟสและจุดเช็คพอยต์

**เงื่อนไขการทำงาน:** รันโปรโตคอลนี้ทันทีหลังงานสุดท้ายในเฟสของ `TODO.md` เสร็จสิ้น

1. **ประกาศเริ่มโปรโตคอล:** แจ้งให้ผู้ใช้ทราบว่าเฟสนี้เสร็จสมบูรณ์แล้ว
2. **ตรวจสอบความสมบูรณ์ทางภาพ (Visual Audit):**
   - รัน Tool `export_mermaid` หรือ `export_canvas` เพื่อดูผังรวม
   - ตรวจสอบไฟล์ที่เปลี่ยนแปลงในเฟสนี้ด้วย `git diff`
3. **รันการทดสอบโครงสร้างแบบเข้มข้น:** รัน `validate_story_structure` แบบ `strict: true`
4. **เสนอแผนการตรวจสอบด้วยตนเอง (Manual Verification Plan):**
   - สร้างแผนการตรวจสอบพล็อตทีละขั้นตอนตามฟอร์แมต:

   **ตัวอย่างแผนการตรวจสอบโครงเรื่อง:**
   ```text
   โครงสร้างอัตโนมัติผ่านแล้ว สำหรับการตรวจสอบด้วยตนเอง โปรดทำตามขั้นตอนดังนี้:

   **ขั้นตอนการตรวจสอบด้วยตนเอง:**
   1. **เปิดไฟล์แผนผัง:** `mermaid-output.md` เพื่อดูภาพรวม
   2. **ตรวจสอบเส้นทาง:** ยืนยันว่าปมในฉากที่ 2 ถูกสานต่อในฉากที่ 5 จริงหรือไม่
   3. **ยืนยันตัวละคร:** ตรวจสอบว่าพัฒนาการของตัวละคร A เป็นไปตามเป้าหมายหรือไม่
   ```

5. **รอการยืนยันจากผู้ใช้:** ถามผู้ใช้ว่าพล็อตสมบูรณ์หรือไม่ และหยุดรอคำยืนยัน "ใช่"
6. **สร้าง Checkpoint Commit:** Stage การเปลี่ยนแปลงและ Commit เช่น `conductor(checkpoint): Checkpoint end of Phase X`
7. **แนบรายงานด้วย Git Notes:** แนบรายงานการตรวจสอบและคำยืนยันของผู้ใช้เข้ากับ Checkpoint Commit
8. **บันทึก Checkpoint SHA ใน TODO.md:** อัปเดตหัวข้อเฟสด้วยรูปแบบ `[checkpoint: <sha>]`
9. **Commit การอัปเดต TODO.md:** Commit ด้วยข้อความ `conductor(plan): Mark phase '<PHASE NAME>' as complete`

## เกณฑ์การควบคุมคุณภาพ (Quality Gates)

- [ ] โครงเรื่องผ่านการตรวจสอบแบบ `strict` จาก Validator
- [ ] ตัวละครทุกตัวถูกบันทึกลงใน Graph ครบถ้วน
- [ ] มีการระบุ **Type** และ **Label** ชัดเจนใน `TODO.md`
- [ ] แผนผัง Mermaid/Canvas แสดงเส้นเรื่องที่ต่อเนื่อง ไม่มีการขาดตอน
- [ ] ฟังก์ชันหรือ Tool ใหม่ต้องมี Zod Schema กำกับเสมอ
- [ ] รายงานจาก Dashboard ระบุว่าปมขัดแย้งมีการไต่ระดับที่เหมาะสม

## ประเภทและป้ายกำกับ (Types & Labels)

**ประเภท (Types):**
- `feat`: เพิ่มเนื้อหาใหม่/ฉากใหม่
- `fix`: แก้ไขจุดขัดแย้งของพล็อต (Plot Hole)
- `plot`: วางโครงสร้างเหตุการณ์หลัก
- `arc`: พัฒนาเส้นทางเดินของตัวละคร
- `style`: ปรับปรุงบทบรรยาย/สำนวน

**ป้ายกำกับ (Labels):**
- `core`: แกนหลักของเรื่อง ห้ามเปลี่ยน
- `high-priority`: งานสำคัญที่ต้องทำทันที
- `character`: งานที่เน้นตัวบุคคล
- `world-building`: งานที่เน้นกฎของโลกและสภาพแวดล้อม

## แนวทางการ Commit (Commit Guidelines)

### รูปแบบข้อความ (Message Format)
```
<type>(<scope>): <description>

[รายละเอียดการเปลี่ยนแปลงโครงเรื่อง]

[SHA ของงานที่เกี่ยวข้อง]
```

---

## นิยามของคำว่าเสร็จ (Definition of Done)

1. เนื้อหาถูกแปลงเป็น StoryGraph JSON ที่สมบูรณ์
2. โครงสร้างผ่านการ Validate 100%
3. มีการ Export แผนผังออกมาเพื่อให้เห็นภาพรวม
4. บันทึกสรุปงานลงใน Git Notes เรียบร้อย
5. อัปเดตสถานะและ SHA ใน `TODO.md` ครบถ้วนตามรูปแบบ

---

## ขั้นตอนฉุกเฉิน (Emergency Procedures)

### พล็อตขัดแย้งอย่างรุนแรง (Critical Plot Hole)
1. สร้างสาขาเฉพาะเพื่อแก้พล็อต (fix/plot-hole)
2. ระบุจุดที่ขัดแย้งใน Graph
3. ปรับปรุงเนื้อหาจนกว่า Validator จะผ่าน
4. ตรวจสอบผลกระทบต่อฉากถัดไปทั้งหมด
5. บันทึกวิธีแก้ลงใน `TODO.md`

---

## การปรับปรุงอย่างต่อเนื่อง (Continuous Improvement)

- รีวิวกระบวนการวางพล็อตทุกสัปดาห์
- อัปเดต Zod Schema ตามความซับซ้อนของเรื่องที่เพิ่มขึ้น
- บันทึกบทเรียนเรื่องการวางโครงสร้างที่ผิดพลาดเพื่อไม่ให้เกิดซ้ำ
